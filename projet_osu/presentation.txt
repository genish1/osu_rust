===============================================================
  PRÉSENTATION DU PROJET : osu!simple
  Jeu de rythme en Rust avec le moteur Bevy
===============================================================


────────────────────────────────────────────────────────────────
 1. INTRODUCTION
────────────────────────────────────────────────────────────────

osu!simple est un jeu de rythme inspiré du jeu en ligne "osu!".
Le principe : des cercles apparaissent sur l'écran en rythme avec
la musique, et le joueur doit cliquer dessus au bon moment.

Le projet a été développé en Rust avec le moteur de jeu Bevy 0.18,
dans un environnement WSL2 (Windows Subsystem for Linux).

Taille du projet : ~1500 lignes de code Rust réparties en 8 modules.


────────────────────────────────────────────────────────────────
 2. POURQUOI RUST ET BEVY ?
────────────────────────────────────────────────────────────────

Rust :
  - Langage compilé, très performant (proche du C++)
  - Garanties mémoire à la compilation : pas de segfault, pas de
    data race, pas de null pointer sans gestion explicite
  - Idéal pour un jeu où la performance et la fiabilité comptent
  - Intéressant à apprendre pour comprendre la gestion mémoire

Bevy :
  - Moteur de jeu open-source moderne écrit en Rust
  - Utilise l'architecture ECS (Entity-Component-System)
  - Gestion intégrée : rendu 2D, audio, fenêtre, gestion d'états
  - Communauté active, bonne documentation


────────────────────────────────────────────────────────────────
 3. L'ARCHITECTURE ECS — LE CŒUR DE BEVY
────────────────────────────────────────────────────────────────

L'ECS est une façon de structurer un jeu très différente de l'OOP
classique. Au lieu d'avoir des objets avec des méthodes, on a :

  Entity    = un identifiant unique (ex : un cercle sur l'écran)
  Component = une donnée attachée à une entité (ex : position, temps)
  System    = une fonction qui traite toutes les entités ayant
              certains composants (ex : "déplace tous les cercles")

Exemple concret dans notre jeu :
  - L'entité "cercle" a les composants :
      HitCircle { time_ms: 2000 }   ← quand le cliquer
      Sprite { image: hitcircle.png }← comment l'afficher
      Transform { x: 400, y: 300 }  ← où l'afficher
      Visibility::Hidden             ← visible ou non

  - Le système update_circles() parcourt toutes les entités
    qui ont HitCircle et Visibility, et les rend visibles
    au bon moment.

Avantage : les systèmes sont indépendants et faciles à tester.
Inconvénient : courbe d'apprentissage importante au départ.


────────────────────────────────────────────────────────────────
 4. STRUCTURE DU CODE — LES 8 MODULES
────────────────────────────────────────────────────────────────

  main.rs       Point d'entrée. Crée l'application Bevy, enregistre
                tous les systèmes et les états.

  beatmap.rs    Parseur du format .osumap personnalisé.
                Lit le fichier texte et construit la structure Beatmap.

  hitobject.rs  Types de données : HitObject (Circle ou Slider),
                avec leur position et timestamp.

  game.rs       GameState : score, combo, max_combo, misses.
                Méthodes : evaluate_timing(), register_hit(),
                max_possible_score(), grade().

  renderer.rs   Systèmes Bevy pour l'affichage :
                - spawn_circles : crée les entités au démarrage
                - update_circles : anime les approach rings,
                  gère la visibilité, supprime les expirés
                - Curseur custom, affichage score, fin de map

  audio.rs      MusicTimer (temps écoulé en secondes).
                Lance la musique, incrémente le timer chaque frame.
                Fonctions pause/resume pour le menu pause.

  input.rs      Gestion des clics (souris + touches X et C).
                Détection de la note la plus proche (<50px),
                évaluation du timing, gestion des sliders.

  menu.rs       Tous les écrans : Menu principal, Sélection de map,
                Décompte, Pause, Écran de résultats.
                Gestion des états (AppState) et des transitions.


────────────────────────────────────────────────────────────────
 5. FORMAT DE BEATMAP PERSONNALISÉ (.osumap)
────────────────────────────────────────────────────────────────

Plutôt que d'utiliser le format .osu officiel (complexe, plusieurs
centaines de paramètres), on a créé un format simplifié :

  title = Level 1
  audio = level1.ogg
  bpm = 128

  circle | 400 | 300 | 1500       ← x | y | temps en ms
  circle | 700 | 200 | 3000
  slider | 200 | 400 | 5000 | 600 | 400 | 6500
                              ↑ x_fin | y_fin | temps_fin

Le parseur lit les lignes, ignore les commentaires (#),
distingue les métadonnées (clé = valeur) des objets (|).
Les objets sont triés chronologiquement à la fin du parsing.


────────────────────────────────────────────────────────────────
 6. SYNCHRONISATION AUDIO / VISUEL
────────────────────────────────────────────────────────────────

Problème : comment savoir à quel moment de la musique on est,
pour afficher les notes au bon endroit ?

Solution : MusicTimer est une ressource qui stocke le temps
écoulé depuis le début de la musique (en secondes).
Chaque frame, update_timer() ajoute time.delta_secs().

Tous les systèmes utilisent ce timer pour savoir "où on en est" :
  elapsed_ms = timer.0 * 1000.0

L'approach ring d'un cercle à time_ms = 3000ms apparaît à
  3000 - 1500 = 1500ms
et met 1500ms à se refermer sur le cercle.


────────────────────────────────────────────────────────────────
 7. SYSTÈME DE SCORE ET DE NOTATION
────────────────────────────────────────────────────────────────

Précision du clic (delta = |temps_clic - time_ms|) :
  Hit300  ≤ 200ms    300 points × combo
  Hit100  201-400ms  100 points × combo
  Hit50   401-450ms   50 points × combo
  Miss    > 450ms      0 point, combo remis à 0

Le combo multiplie les points : frapper 10 notes d'affilée
donne 300 × 10 = 3000 points pour la dernière.

Score maximal théorique (toutes notes en Hit300, combo parfait) :
  max = 300 × n × (n+1) / 2   où n = nombre de notes

Notes finales :
  S  → score == max (perfection absolue)
  A  → ≥ 80% du max
  B  → ≥ 60% du max
  C  → ≥ 40% du max
  D  → < 40% du max

────────────────────────────────────────────────────────────────
 9. DIFFICULTÉS RENCONTRÉES
────────────────────────────────────────────────────────────────

1. Apprendre l'architecture ECS de Bevy
   → Très différente de la POO. Les données et la logique sont
     séparées. Il faut penser "quels composants cette entité a-t-elle"
     plutôt que "quelle classe est-ce que je vais créer".

2. La version de Bevy (0.18) est récente
   → L'API change beaucoup entre versions (0.14 → 0.15 → 0.18).
     Beaucoup d'exemples en ligne sont obsolètes.
     Ex : ImageBundle (ancienne API) remplacé par ImageNode + Node.

3. Le curseur custom n'était pas visible
   → Problème : le curseur était un Sprite 2D, mais les menus sont
     des éléments UI. En Bevy, l'UI s'affiche toujours AU-DESSUS
     des sprites 2D. Le curseur était donc caché par le menu.
   → Solution : transformer le curseur en élément ImageNode (UI)
     avec un GlobalZIndex élevé pour qu'il passe par-dessus tout.

4. Les sliders apparaissaient dès le début de la map
   → Problème : toutes les entités étaient spawnées au démarrage
     sans contrôle de visibilité.
   → Solution : spawner avec Visibility::Hidden, puis passer à
     Visibility::Visible quand l'approach ring doit apparaître.
     Attention : Bevy interdit d'avoir plusieurs Query<&mut Visibility>
     dans le même système sans filtres Without<> pour indiquer
     que les ensembles d'entités sont disjoints.

5. Contraintes de WSL2
   → WSL2 ne supporte pas Vulkan/OpenGL nativement.
   → Nécessité de forcer le rendu software (LIBGL_ALWAYS_SOFTWARE=1)
     et de désactiver Wayland (WAYLAND_DISPLAY="").
   → Conséquence : performances réduites, latence légèrement plus haute
     que sur un vrai Linux ou Windows natif.

6. Latence de la souris
   → Avec le VSync activé, Bevy attendait le prochain refresh
     moniteur avant d'envoyer la frame, ajoutant ~16ms de latence.
   → Solution : PresentMode::AutoNoVsync (Bevy envoie la frame
     dès qu'elle est prête, sans attendre).
   → Plein écran borderless : évite la couche du compositeur de
     fenêtres qui ajoute sa propre latence.

7. Conflits de requêtes Bevy (Query conflicts)
   → Dans update_circles(), on voulait modifier Visibility sur
     plusieurs types d'entités (HitCircle, SliderStart, etc.).
   → Bevy détecte un conflit potentiel si deux Query<&mut Visibility>
     coexistent dans le même système.
   → Solution : ajouter des filtres Without<> pour prouver à Bevy
     que les ensembles sont disjoints.
     Ex : Query<..., Without<HitCircle>>


────────────────────────────────────────────────────────────────
 10. CHOIX TECHNIQUES IMPORTANTS
────────────────────────────────────────────────────────────────

Format beatmap personnalisé
  → Le format .osu officiel est très complexe (centaines de
    paramètres). Notre format minimaliste est lisible et extensible.
    Il pourrait évoluer pour supporter courbes, spinners, etc.

Temps écoulé comme référence (pas de Bevy Timer)
  → On utilise un simple f32 incrémenté chaque frame plutôt que
    le système de Timer de Bevy. Plus simple à interroger depuis
    n'importe quel système avec Res<MusicTimer>.

Spawn de toutes les entités en une fois vs dynamique
  → On choisit de tout spawner au départ (caché) plutôt que de
    spawner dynamiquement chaque note à son heure.
  → Avantage : pas de jank/freeze quand une note apparaît.
  → Inconvénient : une beatmap très longue (des milliers de notes)
    serait lente à spawner. Pour ce prototype, c'est acceptable.

États (AppState) pour la navigation
  → Bevy fournit un système d'états (States) propre.
    Chaque état a ses propres systèmes OnEnter/Update/OnExit.
    Transitions : MainMenu → MapSelect → Countdown → Playing
                                                    ↕ (Pause)
                                              ResultScreen

 11. DÉMONSTRATION
────────────────────────────────────────────────────────────────

Points à montrer lors de la démo :

  1. Menu principal → sélection de map
  2. Décompte 3, 2, 1, GO!
  3. Jeu en cours :
     - Approach rings qui arrivent progressivement
     - Cliquer sur un cercle (Hit300 / Hit100 / Miss)
     - Slider (clic + maintien)
     - Score et combo en temps réel
  4. Menu pause (Escape) → Reprendre, Rejouer, Quitter
  5. Fin de map → écran de résultats avec note

Raccourcis :
  Clic gauche / X / C  → cliquer une note
  Escape               → pause / reprendre

────────────────────────────────────────────────────────────────
 13. CONCLUSION
────────────────────────────────────────────────────────────────

Ce projet nous a permis de :
  - Pratiquer Rust dans un contexte réel et ludique
  - Comprendre l'architecture ECS (Entity-Component-System)
  - Travailler avec un moteur de jeu moderne (Bevy)
  - Gérer les contraintes d'un environnement spécifique (WSL2)
  - Itérer sur un prototype : chaque feature construite sur la
    précédente (parser → affichage → input → audio → menus)

Le résultat est un jeu jouable, avec menus, scoring, notation
et plusieurs maps, développé entièrement en Rust.
